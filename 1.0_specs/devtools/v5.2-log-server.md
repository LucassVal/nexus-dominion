# SPEC: ND-Logger — Console Log Collector for Nexus Dominion v5.2

## SDD (Spec-Driven Development)

---

## PROBLEMA

Eruda captura console no browser, mas os logs morrem na sessao. Nao ha:
- Persistencia de logs entre sessoes
- Acesso do Hermes Agent aos erros de runtime
- Historico de crashes para debug remoto

## SOLUCAO

Servidor HTTP local que coleta logs do jogo em tempo real e salva em arquivo.
Hermes Agent le o arquivo de log e reporta erros sem precisar de copy/paste.

---

## FEATURE 1: LOG SERVER (Backend — Python)

### Data
```
LOG_SERVER = {
  host: '127.0.0.1',
  port: 8099,
  endpoints: {
    '/log':     POST — recebe entrada de log (JSON),
    '/health':  GET  — ping de saude,
    '/flush':   POST — força flush do buffer
  },
  log_file: '~/nexus-dominion/logs/nd-runtime.log',
  max_size:  2MB,           // rotaciona ao atingir
  rotation:  3,             // mantem 3 arquivos
  buffer_size: 10,          // flush a cada N mensagens
  flush_interval: 5000      // flush a cada N ms
}
```

### Mechanics
1. Servidor Python `log_server.py` usando `http.server` built-in (zero deps)
2. Endpoints:
   - `POST /log` — recebe `{"level":"error","msg":"...","stack":"...","ts":123456,"line":42,"file":"..."}`
   - `GET /health` — retorna `{"status":"ok","logs":123,"size":45678}`
   - `POST /flush` — força write do buffer pro disco
3. Buffer ring de 10 entradas em memoria
4. Flush automatico a cada 5s ou quando buffer cheio
5. Rotacao de log: `nd-runtime.log` → `nd-runtime.1.log` → `nd-runtime.2.log` (max 3)
6. Formato de saida: one JSON per line (JSONL)

### Integration
- Iniciar com: `python3 ~/nexus-dominion/server/log_server.py &`
- Parar com: `kill $(cat ~/nexus-dominion/server/log_server.pid)`
- Make target: `make log-server` (start), `make log-server-stop` (stop)

## FEATURE 2: CONSOLE INTERCEPTOR (Frontend — JS)

### Data
```
LOG_INTERCEPTOR = {
  enabled: bool,            // ativado automaticamente se servidor responder
  server_url: 'http://127.0.0.1:8099',
  levels: ['log','warn','error','info','debug'],
  batch_size: 5,           // envia em lotes
  batch_interval: 3000,    // ou a cada 3s
  max_queue: 100,          // descarta se acumular >100
  include_stack: true,     // stack trace para erros
  game_paused: false       // nao envia durante pausa
}
```

### Mechanics
1. Intercepta `console.log`, `console.warn`, `console.error`, `console.info`, `console.debug`
2. Salva chamada original e wrappeia com `apply()`
3. Cada entrada capturada como:
   ```json
   {
     "level": "error",
     "msg": "TypeError: G.cities is undefined",
     "stack": "at gameTick (line 472)\n at setInterval...",
     "ts": 1782706800000,
     "line": 472,
     "file": "nexus-dominion.html",
     "game_day": 42,
     "game_tick": 128
   }
   ```
4. Fila de envio: acumula `batch_size` entradas ou espera `batch_interval`
5. Envia via `fetch(POST /log)` com `keepalive: true`
6. Se servidor offline: armazena em localStorage ring buffer (ultimas 50)
7. Reconecta quando servidor volta (health check a cada 10s)

### Integration
- Carregado pelo loader.js ANTES do jogo iniciar
- Hook nos consoles globais
- Nao afeta output normal do Eruda (pass-through)

## FEATURE 3: LOG VIEWER — ERUDA PLUGIN (Frontend)

### Data
```
LOG_VIEWER = {
  name: 'ND-Logs',
  max_display: 200,
  filter: 'all' | 'error' | 'warn' | 'info',
  auto_scroll: true,
  server_status: 'offline' | 'connected' | 'error'
}
```

### Mechanics
1. Plugin Eruda: aba "ND-Logs" no painel
2. Mostra:
   - Status do servidor (conectado/offline)
   - Contagem de logs enviados
   - Tamanho do arquivo de log no disco
   - Lista dos ultimos 200 logs (com filter)
   - Botoes: Clear, Download, Pause/Resume
3. Cada linha: timestamp + level colorido + mensagem
4. Erros em vermelho, warns em amarelo, info em azul
5. Click em erro expande stack trace

### UI
- Topo: barra de status (servidor, contagem, tamanho)
- Filtro: botoes All | Errors | Warnings | Info
- Lista rolavel com scroll automatico
- Rodape: botoes Clear | Download .log | Pause

## FEATURE 4: HERMES LOG MONITOR (Backend — Python)

### Data
```
HERMES_MONITOR = {
  check_interval: 30,      // segundos entre verificacoes
  alert_on: ['error', 'uncaught_exception', 'crash'],
  max_alerts: 5,           // maximo de alertas por sessao
  report_to: 'terminal',   // onde reportar (terminal, telegram, file)
  ignore_patterns: []      // regex para ignorar (ex: warnings conhecidos)
}
```

### Mechanics
1. Script `monitor_logs.py` que Hermes pode chamar:
   - Le o arquivo `nd-runtime.log`
   - Filtra por entradas novas (usa `last_position`)
   - Extrai erros e warnings
   - Reporta no terminal (ou via Telegram se configurado)
2. Mantem cursor de ultima leitura em `~/.hermes/log_cursor.json`
3. Comando Hermes: `python3 ~/nexus-dominion/server/monitor_logs.py`
4. Output: lista de erros encontrados desde ultima verificacao

### Integration
- Cron job opcional: `make cron-log-monitor` → verifica a cada 5min
- Comando rapido: `make log-check` → verifica agora
- Alerta no Telegram se `deliver='telegram'`

### Output exemplo
```
[ND-LOG] 3 new errors since last check:
  L472: TypeError: Cannot read properties of undefined (reading 'cities')
    at gameTick (nexus-dominion.html:472:23)
  L308: ReferenceError: RNG is not defined
    at cbtAtk (nexus-dominion.html:308:15)
  L127: SyntaxError: Invalid regular expression
    at <script> (nexus-dominion.html:127:45)
```

## FEATURE 5: AUTO-START SERVER WITH GAME (UX)

### Data
```
AUTO_START = {
  enabled: true,
  method: 'python3',
  server_script: '~/nexus-dominion/server/log_server.py',
  health_check_url: 'http://127.0.0.1:8099/health',
  start_timeout: 3000      // ms para aguardar server subir
}
```

### Mechanics
1. `start.sh` script bash:
   - Inicia log_server.py em background
   - Aguarda health check OK
   - Abre o jogo no navegador
   - Printa "Server OK — logs em ~/nexus-dominion/logs/"
2. Make target: `make dev` → start server + open game
3. Make target: `make dev-stop` → kill server

## IMPLEMENTATION ORDER

1. **Fase A**: `log_server.py` — servidor HTTP com buffer + rotacao (Backend)
2. **Fase B**: `console_interceptor.js` — hook nos consoles + envio fetch (Frontend)
3. **Fase C**: `nd-logs.js` — plugin Eruda para visualizacao (UI)
4. **Fase D**: `monitor_logs.py` — leitor de logs para Hermes (Backend)
5. **Fase E**: `start.sh` + Makefile targets (UX)
6. **Fase F**: Injecao no HTML + teste end-to-end

## SAVE/LOAD COMPATIBILITY

Nao afeta game state. Logs sao externos. Interceptor pode ser desligado setando `LOG_INTERCEPTOR.enabled=false`.
